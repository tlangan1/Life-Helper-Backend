# Production Policy for Copilot-Assisted Development

## Purpose

This policy defines how AI assistance is used in production software work. The goal is to improve delivery speed while preserving quality, security, and maintainability.

## Scope

This policy applies to all code, schema, scripts, tests, and documentation produced with Copilot assistance.

## Ownership Rules

1. Engineers own all final technical decisions and shipped code.
2. Copilot output is a draft, not an authority.
3. Architecture, security boundaries, and business-critical logic remain human-owned.

## Allowed Uses

1. Boilerplate generation.
2. Refactoring with tests.
3. Test creation and expansion.
4. Documentation and migration helpers.
5. Small scoped changes to existing files.

## Restricted Uses

1. Shipping Copilot-generated code without review.
2. Large rewrites without staged verification.
3. Prompting with secrets, credentials, private keys, or protected personal data.
4. Unreviewed changes in authentication, authorization, encryption, payments, or data-access controls.

## Required Workflow

1. Define scope first: objective, constraints, and acceptance criteria.
2. Generate changes in small increments.
3. Validate each increment locally.
4. Require tests for behavior changes.
5. Open a reviewable PR with rationale.

## Prompting Standards

1. Include language, framework, and version constraints.
2. State expected behavior and edge cases.
3. Ask for minimal diffs over full rewrites.
4. Request explicit error handling and null handling where relevant.
5. Ask Copilot to follow existing repository patterns.

## Quality Gates

A PR using Copilot must pass all of the following:

1. Build, lint, and type checks pass.
2. Unit and integration tests pass.
3. New or updated tests exist for changed behavior.
4. No unresolved security findings from CI scanners.
5. Reviewer confirms readability and maintainability.

## Security Requirements

1. Never paste secrets into prompts.
2. Use least-privilege patterns for data access.
3. Validate and sanitize inputs.
4. Prefer parameterized SQL and safe query construction.
5. Require explicit review for high-risk modules.

## Review Checklist

1. Is the change small enough to review safely?
2. Are edge cases covered?
3. Are error paths and rollback behavior correct?
4. Are naming and patterns consistent with the codebase?
5. Is there any hidden behavior change?
6. Are tests meaningful, not only happy-path?

## Incident Rule

If a production issue is traced to AI-assisted code:

1. Create a post-incident note.
2. Add a regression test.
3. Update this policy or prompt templates to prevent recurrence.

## Metrics

Track these metrics monthly:

1. PR cycle time.
2. Review rework rate.
3. Escaped defects.
4. Test coverage delta on AI-assisted PRs.
5. Security findings per PR.

## Enforcement

1. Reviewers may reject PRs that do not meet this policy.
2. Exceptions require documented approval in the PR description.

## Suggested PR Note Template

- Copilot used: Yes or No
- Scope generated by Copilot: Files and functions
- Human validation performed: Build, tests, manual checks
- Risk level: Low, Medium, High
- Security-sensitive areas touched: Yes or No
- Why this is safe to merge: Short rationale
